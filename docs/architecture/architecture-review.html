DRF の request と graphene の info.context が、1 つの uri-builder シームに対する 2 つの実アダプターになる。graphql→serializers の依存辺は消える。

問題。「このオブジェクトの公開アイデンティティとは何か」は 1 つの概念だが、インターフェースを持たないためサブシステムごとに再実装されている。

解決。urn.model_urn を小さな ModelIdentity モジュールへ昇格させ、urn・endpoint・display と絶対 URI を所有させ、uri-builder を抽象化する。

  • レバレッジ
    1 つのインターフェースを REST と GraphQL の両方が呼ぶ
  • 局所性
    urn/endpoint の形式変更が 1 か所で済む
  • テスト
    純粋関数。Field のライフサイクルも resolver も不要

シームを正当化する 2 つのアダプター：REST の request.build_absolute_uri、GraphQL の info.context.build_absolute_uri。このシームは仮想ではなく実在する。

5 · 削除テストを本気でやる。これらのモジュールは消える

強く推奨 深化ではなく削減

スキルが定める判定。モジュールを削除してみて、複雑さが消えるなら、それはパススルーだった。 ここに挙げたいくつかは完全に消える。呼び出し元もテストもない。正直な「深化」は削除であり、それが残りを研ぎ澄ます。

さらに、深化提案がいずれも見落とした潜在バグ。 DownloadMixin.response_field_data（viewsets.py:111–121）は create_download_filefield_response(…, format=format) を呼ぶが、format はこのメソッドの 引数ではなく、Python のビルトインに解決される。ストレージダウンロード経路は format が truthy だと例外になる。 ダウンロード経路にはテストが 1 つもないため、これまで表面化しなかった。

これが最も価値の高い発見である理由。このライブラリの本当の摩擦は浅いモジュールではなく、推測的に肥大化した表面積だ。パススルーを削除し、生きているシーム（filters・downloads・batch・urn）に振る舞いのテストを足すほうが、どの単一の深化よりもナビゲーション性とテスト容易性に効く。

最優先の推奨

まず drf-spectacular アダプター、次に アイデンティティ を深化させる。

spectacular アダプターは、明確に「集約であって移動ではない」最もリスクの低い一手だ。実在する 2 アダプターの シーム（インストール済みか否か）で、すでにドリフトしており、スタブを一度だけ定義することで直る。シームを扱う 習慣づけのきっかけになる。要となる深化はアイデンティティだ。重複した概念を真に深いモジュールへ変え、 同時にレイヤリングの逆転（graphql が serializers を逆参照している状態）を取り除く唯一の候補で、uri-builder の 実アダプターが 2 つある。候補 5（消えるモジュールを削除し、生きたシームをテストする）は、その両者をつなぐ 結合組織として扱う。ここがライブラリの局所性を最も稼げる場所だ。

あえて最初には推奨しないもの。ネストフィールドの「戦略」書き換え（複雑さを移動させるだけ。孤児削除シームは すでに機能している）と、WordFilter／clone_filter_fields のビルダー（呼び出し元が 1 つ、仮想シーム）。 敵対的レビューは 3 つの大きな抽象化版をすべて却下した。
